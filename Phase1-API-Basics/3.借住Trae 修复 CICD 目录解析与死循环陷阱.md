# 📝 Phase 1 实战复盘：借力 Trae 修复 CI/CD 目录解析与死循环陷阱

**日期**：2026-05-08

**项目归属**：AI 应用开发学习路线 - Phase 1 (工具链与 AI 编程初探)

**协同工具**：Trae (Builder 模式)、GitHub Actions、Python

## 一、 遇到的问题

在搭建我的独立阅读仓库（`Reading-and-Comprehension`）时，我希望通过 GitHub Actions 和 Python 脚本，实现“只需 push 笔记，README 目录自动生成”的 CI/CD 闭环。

原本以为只是一个简单的文件遍历脚本，但在实际应用中，却触发了一系列令人崩溃的连锁 Bug。以下是我遇到的核心问题清单：

1. **出现大量重复的文件名**：由于正则替换逻辑的缺陷（占位符变量为空，或者受隐藏换行符干扰），脚本找不到替换边界，导致每次运行都在 README 里重复追加目录，直接塞出几十个重复块。
2. **特殊符号引发的 Markdown 崩盘**：我的笔记命名为 `阅读笔记：[参考模板].md`。Markdown 解析器一看到中括号 `[]` 就直接大脑宕机，无法识别这是个超链接，导致渲染出纯文本乱码。
3. **首行一空，标题就没了**：最初的代码使用 `readline().strip()` 提取文章内部的 `# 标题`。一旦文件的第一行是空行或图片，代码就会抛出异常，全部回退成光秃秃的文件名。
4. **占位符错位**：生成的目录总是不在预期位置，排版混乱，后来发现是因为自己在 `README.md` 中放置 `` 标签的位置不对。

## 二、 引入 Trae 

连环 Bug 已经铺了一层，我不想再靠搜索引擎零碎地扒帖子了。我把出问题的 `github.py` 脚本抽出来，丢到我自己那个 Phase 1 的学习仓库里当“试验场”，全程开着 Trae（Builder 模式）协同改。修复过程没走常规搜索路线，但中间也被 AI 结结实实坑了两次。

### 1. 见识 AI 的陷阱 (Context Loss)

当我把文件名包含特殊字符导致解析错误的问题喂给 Trae 时，它迅速给出了使用 `urllib.parse.quote()` 进行 URL 编码的方案。

**防 AI 忽悠：**

- Trae 为了解决符号问题，非常暴力地使用了 `re.sub(r'[^\w\s-]', '', filename)`，这直接把我标题里的中文全删了。
- URL 编码虽然让系统不报错了，但把原本可读的中文路径变成了 `%E9%98%85%E8%AF%BB...` 这种极度难看的机读乱码。
- 更致命的是，Trae 擅自删减了我原代码中读取文件业务逻辑的代码，给了一个“阉割版”的示例。

### 2. 从“帮我写”到“按我说的做”

我意识到，**不能把 Trae 当作写代码的黑盒，而要把它当成一个算力极强但缺乏全局观的打字员。**

我重新调整了 Prompt（提示词），明确禁止它使用暴力的正则和 URL 编码，强制要求寻找 Markdown 原生的优雅解法。最终我们在协同中确定了最佳实践：

- **对付标题括号**：弃用转义符 `\`，改用 HTML 实体编码 `[` 和 `]` 替换 `[` 和 `]`，在视觉上完美呈现括号，同时彻底避开底层语法树的解析。
- **对付奇葩路径**：弃用 URL 编码，利用 GitHub Flavored Markdown (GFM) 隐藏的**尖括号包裹法** `<文件路径>`，完美兼容中文、全角冒号和空格。

## 三、 最终方案：

为了彻底消灭正则带来的“无限追加”问题以及“首行读取失败”的问题，我在 Trae 的协助下对代码进行了大重构，实现了两项“暴力美学”级别的防御：

1. **只认文件名，不读文件**：不再去读取文件内部，直接将 Markdown 文件的名称（去除 `.md`）作为最终生成的目录标题。所见即所得。
2. **“物理切片法”替换正则**：抛弃不可控的正则表达式，直接使用 Python 的 `string.find()` 定位 `` 的物理索引位置。像切蛋糕一样把中间的旧目录彻底挖空，然后填入新目录。**这从数学逻辑上 100% 杜绝了重复写入的可能。**

## 四、 所学

1. **AI 辅助编程的真相**：Trae 接入 API 和不接入 API 在我这次使用中体感一致，这让我深刻体会到了优秀软件工程的“封装抽象”魅力。不管后端调用的模型是深是浅，前端的工程体验是一致的。这也启发了我未来学习 API 调用的业务方向——我是要亲手做一个类似 Trae 的应用端。
2. **脏环境里别乱动**：永远不要在脏数据堆满的生产环境（真仓库）里乱改。在独立文件夹里新建空壳文件复现 Bug，修复后再移植，这是极其重要的工程素养。
3. **架构的决断还是得自己下**：AI 写代码很快，但它不懂业务逻辑的边界。出了乱码不要慌，精准定位问题（是正则锚点丢失、还是 AST 解析失败），用工程语言重新写 Prompt 去引导 AI，这才是未来 AI 程序员的核心壁垒。
