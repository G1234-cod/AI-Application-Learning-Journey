## 第一部分：接触资料前的初始认知

在学习这篇文章和视频之前，我脑子里对这些概念的理解大概是这样的：

1.  **关于 Prompt Engineering**
    
    我知道有一些提问套路能让 AI 回答得更好。比如“你是一个资深程序员，请帮我写一个排序函数”，或者在提问时加上“请一步一步思考”。但我没想过这背后有一个完整的工程化演进脉络。
    
2.  **关于 Agent**
    
    我对这个词的理解非常模糊。我一度以为 Agent 要么是一种新的 AI 模型（就像 DeepSeek、GPT 一样），要么就是“用 AI 做的一个项目”。说实话，我完全不知道 Agent 和“模型”到底是不是同一个东西。
    
3.  **关于 Spec 和需求文档**
    
    我心里想：Spec 不就是传统软件工程课上学的那种“需求规格说明书”吗？告诉开发者该写什么功能的。我觉得它和 AI 的关系，无非就是把需求文档喂给 AI 看，让它照着写代码。

4.  **关于工具对比**
    
    我的思维模式很简单：什么工具给的结果好、效率高，就用什么。Spec-Kit 和 OpenSpec，不就是两个 AI 编程助手吗？哪个好用用哪个就行了。

**客观地讲，以上这些初始认知大部分是偏的、混的、或者停留在表层的。**

## 第二部分：认知冲突与追问过程

看完文章和视频后，我没有直接接受里面的结论，而是主动把脑子里冒出来的矛盾点一个一个挑了出来，然后死磕到底。下面是我追问的全过程。

### 冲突 1：大模型应用的演进——为什么会有从 Prompt 到 Harness 的四个阶段？

- **我看到的内容**：文章和视频都提到，人们在使用大模型时，经历了一个从“优化提问”到“给模型装上手和脚”的演进过程。

- **我的疑惑**：这些阶段到底在解决什么问题？为什么不能一步到位？

- **追问后理清的逻辑**：每一个阶段的出现，都是因为上一个阶段暴露了无法解决的硬伤。这不是某个天才突发奇想，而是问题驱动的自然演化。

| 阶段                        | 当时人们遇到的痛點                                           | 给出的解决方案                                               |
| --------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **Prompt Engineering**      | AI 回答质量不稳定，有时胡言乱语                              | 规范输入格式：Few-shot（给示例）、Chain-of-Thought（要求一步步推理） |
| **Context Engineering**     | 模型没有记忆，聊到下轮就“失忆”了                             | 在请求中注入结构化的上下文：RAG 检索增强、AGENTS.md 文件、记忆系统 |
| **Harness Engineering**     | 模型只会“说”，不能“做”。让它写代码，它写完就完了，不能跑、不能测、不能操作文件 | 给模型接上“身体”：文件系统、Shell 执行权限、沙箱隔离环境、工具调用能力 |
| **Spec-Driven Development** | 即使模型有了手脚，但它经常偏离方向。让它优化代码，它可能重写整个功能 | 在动手前先签一份精确的“合同”（Spec），明确数据模型、API 端点、边界条件、验收标准，然后再让 Agent 照章办事 |

**我理解到的关键**：这四个阶段不是互相替代的关系，而是**层层叠加**的关系。一个成熟的 AI 应用（如 Cursor、Copilot）其实同时集成了这四层能力。

---

### 冲突 2：Harness 到底是什么？它比 Spec-Kit 更好吗？

- **我最初的错误理解**：我看到视频里展示 Agent 能在沙箱里跑代码、展示运行结果。我就想：Spec-Kit 让 AI 写完代码后，只给你看一堆代码，人得自己读代码判断好坏。而 Harness 能直接把代码跑起来让你看到实际输出，那肯定是 Harness 比 Spec-Kit 更牛啊！

- **追问后得到的纠正**：这个想法是**完全错的**。我把两个不在同一个层面上的东西放在一起比了。用一个建筑业的比喻来理解：

| 概念                  | 建筑业的类比                                                 | 核心职能                                            |
| --------------------- | ------------------------------------------------------------ | --------------------------------------------------- |
| **Harness**           | 施工队配备的全套装备：电动工具、安全帽、脚手架、安全网、起重机、对讲机 | **提供执行能力和安全边界**。它是 Agent 的“身体”。   |
| **Spec-Kit/OpenSpec** | 建筑设计师和她的全套图纸规范流程                             | **提供规划能力和质量保障**。它是 Agent 的“设计院”。 |

- **关键结论**：

  > - 我之所以能在 Harness 环境下看到代码执行结果，是因为 Harness 这个“装备包”里**包含了一个叫做“沙箱”的封闭测试车间**。Agent 可以在这里面试着跑一下代码，把结果展示给人看。但**是否最终采用这个方案，仍然是人来做的决策**。沙箱只是让“执行—反馈”这个循环变快了。
  >
  > - Spec-Kit 的重点则完全不同——它关心的是**图纸本身有没有逻辑错误**，它在动工前反复审查的是“规格”，而不是去工地上试盖一堵墙。
  > - **所以不能比较谁更好，因为它们解决的是完全不同的问题。** 一个成熟的 Agent 系统，往往同时需要这两样东西。

---

### 冲突 3：“能执行操作”到底是谁的能力？

- **我最初的困惑**：

> - “Spec-Kit 可以像 Harness 那样执行任务吗？”
> - 我搞不清楚，“能操作计算机”这件事，到底是“网页 AI 和软件 AI”的区别，还是“Agent 和别的什么东西”的区别，还是“Harness 和 Spec-Kit”的区别。

- **追问后得到的清晰答案**：这个能力的来源链条是这样的：

  1.  **根本区别：Agent vs. 裸模型**
      -   **裸模型**（如浏览器里直接打开的 ChatGPT 网页版）：只能接收文本，输出文本。它没有文件系统访问权限，不能执行代码，不能操作你的项目文件。
      -   **Agent**（如 Cursor 里集成的 AI、Copilot）：能“做”事情。它能打开你的文件、修改代码、运行终端命令、搜索网页。
  
  
    2. **Agent 之所以能“做”，是因为集成了 Harness**。而Harness 组件里集成了 **工具（Tools）** 和 **沙箱（Sandbox）**。
       -   **文件工具**：让 Agent 能读写项目中的任何文件。
       
       -   **Shell 工具**：让 Agent 能在沙箱里运行 bash 命令。
       
       -   **浏览器工具**：让 Agent 能操控浏览器完成表单填写、网页抓取等任务。
       
       -   **代码解释器**：让 Agent 能执行 Python 等代码片段并获取运行结果。
       
  3. **“网页 AI vs 软件 AI”这个表象下的本质**
  
     -   网页版 AI 通常是**没有集成 Harness 的裸模型**。
  
  
     -   编程软件里的 AI 是**集成了基础 Harness 的 Agent**。

**最终答案**：能执行操作，是 **Agent（因为它有 Harness）** 与 **裸模型** 的区别。SDD 框架（Spec-Kit）本身不执行任何代码，它只是一个“规划工具”，负责在 Agent 动手之前把更清晰的指令传给 Agent。

### 冲突 4：Spec 到底是谁写的？SDD 框架是谁写的？Agent 又在哪里？

- **我最初的混乱**：“Spec 是人写的，还是 SDD 框架写的，还是 AI 写的？SDD 框架是人写的还是 AI 写的？Agent 到底在这堆东西里是什么角色？”

- **追问后拆清的关系**：这是一个静态的组成问题。我把每个东西的“作者”和“角色”彻底分开：

| 东西                                | 谁写的/谁的产物                          | 角色                       | 具体做什么                                                   |
| ----------------------------------- | ---------------------------------------- | -------------------------- | ------------------------------------------------------------ |
| **Spec（规范文件）**                | **由人主导创建**，工具辅助生成           | 一份精确的“合同”           | 包含数据模型、API 端点、边界条件、验收标准。它是人类意图的精确表达。 |
| **SDD 框架（Spec-Kit / OpenSpec）** | **完全是人写的工具软件**                 | 强制推行的自动化开发工作流 | 先通过模板和校验帮人把模糊想法变成结构化 Spec；当 Spec 敲定后，再把这个指令交给下游执行。 |
| **Agent**                           | **人写的框架中，负责执行的那个核心部分** | 执行者/包工头              | 接收 Spec，利用它集成的 Harness 工具去操作文件、运行命令，最终完成代码生成。 |
| **AI 模型（如 DeepSeek、GPT）**     | **模型厂商训练和提供**                   | 大脑                       | Agent 发出的“生成代码”“判断测试结果”等所有核心推理动作，最终都是这个大脑完成的。 |

---

### 冲突 5：传统需求文档不也要给 AI 看吗？Spec 凭什么就是新的？

- **我最初的质疑**：不管是传统 PRD 还是 Spec，不都是一份文档吗？交给 AI 看，AI 照着写代码，有什么区别？

- **追问后得到的核心差异**：这个问题确实问到点子上了。因为最初我也觉得 Spec 就是“需求文档”换个名字。但事实完全不是这样。

> **传统 PRD（产品需求文档）的特征**：
>
> - **目标读者是人**（开发者、PM、测试），不是机器。
>
> - **语言模糊**：“用户登录功能要安全”，什么叫安全？用什么加密？防什么攻击？全靠开发者自己脑补。
>
> - **与代码实现脱节**：写代码的时候可能已经不看 PRD 了，或者 PRD 已经过时了。
>
> - **给 AI 看的后果**：AI 必须“猜测”大量细节，猜错了就偏离方向，回滚和返工的成本极高。
>
> ---
>
> **SDD 中的 Spec 的特征**：
>
> - **目标读者是人 + AI + 工具，三者共同读取**。
> - **语言精确、结构化**：用 Given/When/Then 结构描述行为，明确输入输出的数据模型、API 端点路径、错误处理方式、边界条件。
> - **与实现强绑定**：Spec 是“可执行的生产原料”，AI 可以直接按它生成代码和测试，不再需要猜测。
> - **核心价值**：在写哪怕一行代码之前，人和 AI 就对要构建什么达成了**精确、无歧义的共识**。这极大降低了迭代中的“回滚”和“返工”。

**一个具体的对比示例**：

| 维度               | 传统 PRD                    | SDD 中的 Spec                                                |
| ------------------ | --------------------------- | ------------------------------------------------------------ |
| 对“登录功能”的描述 | “用户可以用邮箱和密码登录”  | `POST /api/login`，请求体 `{"email": string, "password": string}`，密码用 bcrypt 加密，失败 3 次锁定账号 30 分钟，返回 JWT Token |
| 边界条件           | 没写                        | 明确列出：空值处理、超长输入、特殊字符、SQL 注入防护         |
| AI 怎么用          | AI 猜着写，可能用明文存密码 | AI 精确按 Spec 生成代码和测试                                |

## 第三部分：两个主流 SDD 框架的深度对比

文章中花了很大篇幅对比 Spec-Kit 和 OpenSpec，因为这是目前规范驱动开发领域最成熟的两个工具。

### Spec-Kit（GitHub 官方）

-   **出身**：GitHub 官方发布，背靠微软。
-   **设计哲学**：工程化、重流程，像一个严格的“瀑布式”开发流程。
-   **核心工作流**：
    `Constitution → Specify → Clarify → Plan → Tasks → Analyze → Implement`
    每一步都需要人工确认，形成完整的闭环。
-   **宪法（Constitution）**：这是一个全局约束文件，定义项目的最高准则。比如：“本项目使用 TypeScript”、“所有 API 必须有单元测试”、“密码必须用 bcrypt 加密”。它是整个仓库的最高行为准则，所有 Spec 都必须遵守宪法。
-   **适用场景**：从零到一的**绿地项目**，需求明确，需要严格的品质把控。

### OpenSpec（社区工具）

-   **出身**：社区驱动的开源工具。
-   **设计哲学**：轻量、敏捷，像一个敏捷开发的迭代模型。
-   **核心工作流**：`propose → apply → archive`，极其精简，只有三步。
-   **增量变更**：OpenSpec 没有全局宪法，而是围绕一个一个独立的“增量变更”来组织。每一个新功能、每一个 Bug 修复，都是一个独立的 Spec。做完一个就归档一个。
-   **适用场景**：在**存量项目（Brownfield Project）**上迭代新功能。因为它不要求你一次性规划整个项目，而是可以在现有代码基础上“小步快跑”。

### 我理解的技术选型决策逻辑

面对一个实际项目，现在我能这样判断：

| 判断条件                               | 推荐使用     | 原因                                                |
| -------------------------------------- | ------------ | --------------------------------------------------- |
| 从零开始的新项目（绿地）               | **Spec-Kit** | 有全局宪法约束，能从一开始就建立规范，每一步可审查  |
| 300 万行代码的存量项目加新功能（棕地） | **OpenSpec** | 不限入全局约束的坑，每个变更是独立的 Spec，灵活轻量 |
| 小团队快速迭代                         | **OpenSpec** | 流程极简，不绑定 IDE，学习成本低                    |
| 大团队、高安全要求                     | **Spec-Kit** | 门控流程确保每一步都有人确认，可审计                |

## 第四部分：Harness 的核心组件与安全机制

### Harness 六大核心组件

视频中详细拆解了 Harness（以字节跳动的 DeerFlow 2.0 为例），它由六大组件组成，每个组件对应解决裸模型的一个硬伤：

| 组件                  | 解决的裸模型硬伤  | 具体功能                                         |
| --------------------- | ----------------- | ------------------------------------------------ |
| **文件系统**          | 无持久存储        | 管理项目中的所有文件，支持创建、读取、修改、删除 |
| **Bash + 沙箱**       | 不能执行代码      | 在隔离环境中安全运行任意 Shell 命令和代码        |
| **记忆（AGENTS.md）** | 无记忆能力        | 通过结构化文件注入项目上下文，无需训练模型       |
| **Web Search + MCP**  | 知识截止日期限制  | 联网搜索最新信息，通过 MCP 协议调用外部工具      |
| **上下文工程**        | 长对话中信息腐烂  | 维护多轮对话的语义连续性，确保 Agent 不跑偏      |
| **编排 + Hooks**      | 多 Agent 协作混乱 | 编排多个子 Agent 的工作流，保障协同质量          |

---

### 安全沙箱的深度理解

我最关心的是沙箱，因为它直接关系到“AI 会不会搞坏我的电脑”。

-   **本质**：一个与宿主机、内网、其他用户数据**完全隔离的封闭执行环境**。它拥有独立的内核、文件系统和网络栈，攻击面等同于一台独立虚拟机。
-   **作用**：让 Agent 生成的不可信代码在受控空间中运行。即使代码有恶意行为或 Bug，也无法触及真实系统。
-   **我的误解纠正**：沙箱**不是“预览机制”**。Agent 不会先在沙箱里试跑一遍再问你要不要执行。它是直接把代码扔进沙箱跑，而**权限门控**是在沙箱之前就生效的。

---

### 权限模式

Harness 提供了三层权限控制，这是 Agent 安全的最后一道防线：

| 模式       | 行为                                                         | 适用场景                                                     |
| ---------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **ask**    | 每次工具调用前都弹确认框，人点了“允许”才执行                 | 高风险操作：删除文件、修改配置、往生产环境发请求             |
| **auto**   | 自动允许安全操作（如读文件），自动拦截危险命令（如 `rm -rf /`） | 日常开发中的大部分场景                                       |
| **bypass** | 全部自动允许，不弹任何确认                                   | **极其危险**，必须配合严格沙箱使用，一般只在自动化 CI 场景中用 |

---

### MCP 协议

-   **全称**：Model Context Protocol（模型上下文协议）。
-   **作用**：一种标准化协议，让 AI Agent 能**动态发现和调用外部工具**，而不是把所有工具都硬编码在系统里。
-   **与 Function Calling 的区别**：Function Calling 是模型层面的内置机制（模型自己知道该调用什么函数），MCP 是**工具集成层面的标准化协议**（定义工具如何注册、如何被发现、如何被调用）。

## 第五部分：完整的关系调度图

经过所有追问，现在我能在脑子里完整画出整个系统的协作关系了。假设任务是“开发一个用户登录 API”，整个调度过程是这样的：

**第一步：人提出需求**
-   人对系统说：“我要一个安全的登录 API，要能防止暴力破解。”
-   人决定了**做什么**（What）和**验收标准**。

**第二步：SDD 框架介入规划**
-   SDD 框架开始工作，问澄清问题：
    > - “用什么认证协议？JWT 还是 Session？”
    > - “密码如何加密？bcrypt 还是 Argon2？”
    > - “防暴力破解是用验证码还是 IP 限流？”
    > - “API 的 URL 路径是什么？”
    > - “失败几次后锁定？锁多久？”
-   根据人的回答，框架生成一份精确的 `spec.md`，将所有需求定义为可验证的条款。
-   人审核这份 Spec，确认无误。

**第三步：Agent 接手执行**

-   框架将 `spec.md` 正式移交给 Agent。
-   Agent 读取 Spec 后，知道自己要做的事：
    
    > - 创建 `POST /login` 端点。
    > - 密码用 `bcrypt` 加密存储。
    > - 失败 3 次后锁定账号 30 分钟。
    > - 返回 JWT Token。
-   Agent 开始通过 Harness 工具执行：
    > - **调用文件工具**：创建 `auth_controller.py`，写入代码。
    > - **调用沙箱**：运行 `pytest`，检查测试是否通过。
    > - **如果忘了 bcrypt 的用法**：调用搜索引擎工具去查询。

**第四步：AI 模型完成所有推理**

-   Agent 发出的“生成代码”、“判断测试结果”、“搜索信息”等所有核心动作，最终都是由背后的 AI 模型（如 DeepSeek、GPT）完成的。

**用一句话总结这四者的调度关系**：

> **人**制定战略，通过**SDD 框架**将战略转化为精确的战术规划（Spec），然后交由**Agent**利用它的**Harness**（身体）去执行，而**Agent**的每一次思考和行动，都源自其内部的**AI 模型**（大脑）。

## 第六部分：我的认知升级清单

经过这一轮完整的学习和深挖，我的大脑里发生了以下几个关键转变：

1.  **从“比优劣”到“看层级”**：
    -   不再问“Harness 和 Spec-Kit 谁更好”。
    -   而是明白它们是 Agent 大厦里的不同层级：一个是执行基础设施，一个是规划管理工具，各司其职，不冲突。

2.  **从“重结果”到“重契约”**：
    -   认识到在 AI 时代，一份精确、结构化的 **Spec 的价值远大于 AI 临时生成的那一堆代码**。
    -   Spec 是我、AI、团队之间永不偏离的“合同”，是降低协作成本的最高杠杆点。

3.  **从“学概念”到“建系统”**：
    -   能把四个演进阶段串成一条完整的、有逻辑的链条：`提问技巧 → 管理记忆 → 赋予能力 → 建立契约`。
    -   理解了为什么“约束治理（Harness）”和“规划分离（SDD）”是让 Agent 真正成为可靠生产力的两大支柱。

4.  **从“会用”到“会选”**：
    -   理解了 Spec-Kit 和 OpenSpec 背后截然不同的设计哲学后，我能在面对实际项目时，基于“团队规模大小”做出清醒的技术选型，而不是随大流。

5.  **从“表面功能”到“底层安全”**：
    -   理解了沙箱不是预览工具，而是隔离执行环境。
    -   理解了权限模式是 Agent 安全的生命线，bypass 模式是生产环境的大忌。
