## 二、学习过程中遇到的问题 & 如何解决的

### 问题1：为什么每次调用API都要把历史对话全发回去？模型不记事儿吗？

**发现过程**：在写 `terminal-chatbot.py` 时，我最初只发了当前用户输入，结果AI完全不记得上一句说了什么。

**学到的原理**：大语言模型是无状态的。每次API调用都是一次独立的推理，服务器端不保存任何会话历史。所谓的“记住上下文”，其实是客户端在每次请求时，把`[system, user1, assistant1, user2, assistant2, ...]`的全量历史重新发给服务器，让模型“假装”记住。

**解决方案**：维护一个全局的 `messages = []` 列表，每轮都把用户的输入和AI的回复都 `append` 进去，然后每次调用API时传完整的 `messages`。

---

### 问题2：`response.choices[0].message`到底取了什么？为什么是`[0]`？

**发现过程**：看到API返回示例中的 `choices` 是列表，不知道它是不是包含多个候选回复。

**学到的原理**：API的 `choices` 列表长度由参数 `n`（生成数量）决定，默认为 `1` 及只有一条信息，所以一般只有一个元素。`choices[0]` 就是这次调用的唯一完整回复，`.message` 里才包含真正的对话内容（`role` 和 `content`）。

**验证手段**：用 `print(len(response['choices']))` 验证长度确实是1；用 `print(type(response['choices'][0]['message']))` 确认它是字典。

---

### 问题3：`openai`库是什么？既然用它，为什么还要学`requests`？

**发现过程**：DeepSeek官方文档（[Tool Calls | DeepSeek API Docs](https://api-docs.deepseek.com/zh-cn/guides/tool_calls)）既给了 `openai` 库的调用示例，也有 `curl` 的原生请求示例。

**学到的原理**：

- `openai` 库是 OpenAI 官方发布的 Python SDK，DeepSeek 因为兼容OpenAI接口，所以可以直接用这个库。
- SDK的作用：把HTTP请求封装成面向对象的方法，自动处理认证Header、JSON序列化、异常重试等。
- 用 `requests` 手写调用，能让你彻底搞懂大模型API的调用契约：必须发POST到正确的端点，必须在Header里加 `Authorization: Bearer <key>`，必须在Body里传 `model` 和 `messages`。

**最佳实践**：先用 `requests` 跑通，再用 `openai` 提效。

---

### 问题4：`requests`的URL是`https://api.deepseek.com/chat/completions`，而`openai`的`base_url`是`https://api.deepseek.com`，为什么不一样？

**发现过程**：配置 `openai` 库时，填了 `base_url="https://api.deepseek.com"`，调用 `client.chat.completions.create()` 时却不需要写 `/chat/completions`。

**学到的原理**：
- `requests` 需要完整的端点URL，因为它是纯HTTP请求。
- `openai` 库只需要基础地址，它内部会自动拼上固定的功能路径。这是一种常见的设计模式，让调用方更简洁。

---

### 问题5：API Key到底放哪里？怎么避免泄露？

**发现过程**：key关系到AI的连接，直接在代码中体现，会导致被人恶意刷取token

**学到的原理**：

- 密钥绝对不能硬编码在代码中，否则一旦提交到Git仓库，就会全网泄露。
- 本地开发标准做法：使用 `.env` 文件 + `python-dotenv` 加载环境变量，并且把 `.env` 加入 `.gitignore`。
- 生产环境更安全的方式：使用平台的环境变量注入，或者专门的密钥管理服务（KMS）。

**实操**：
1. 创建 `.env` 文件，内容 `DEEPSEEK_API_KEY="sk-xxx"`
2. 安装 `python-dotenv`，在脚本开头调用 `load_dotenv()`
3. 用 `os.environ.get("DEEPSEEK_API_KEY")` 读取
4. 把 `.env` 加入 `.gitignore`

### 问题6：`print(api_key)`有什么风险？

**发现过程**：调试时我为了确认密钥加载成功，在代码里写了 `print(api_key)`。

**学到的原理**：

- 在本地终端打印看似无害，但如果后期把程序改成后台服务，或者把日志重定向到文件，密钥就会以明文形式被记录下来。
- 正确的调试方式：打印密钥的类型和部分内容，如 `print(type(api_key), api_key[:4] + "****")`，看到加载成功且前缀正确即可，不需要看到全部内容。

### 问题7：Token的隐形成本——对话越长，费用越高

**发现过程**：
当我在 `while` 循环中持续对话时，注意到 `messages` 列表会不断增长。我突然意识到——每次请求都带着全量历史，那Token消耗岂不是也在滚雪球？

**我的原始思考**：
我最初的想法是：对话中可能有重复的内容，可以通过**集合去重**来剪枝。然后统计每条信息的出现次数和出现时间，按照一定权重形成一个公式，计算每条信息的“重要性分数”，最后只保留分数高的那部分信息。

**迭代后的认识**：
这个思路有工程价值，但在对话场景下存在根本问题——大模型依赖的是**上下文连贯性**，而不是句子级别的重要性。如果靠集合去重，很可能会丢掉修改过程（比如“把端口改成8080”→“哦，还是用3000吧”），导致语义断裂。当前阶段的正确做法是**滑动窗口裁剪**：只保留最近N轮对话，或者用 `tiktoken` 实时计算Token数，超过阈值时自动删除最早的对话。更高级的做法还包括对早期对话做自动摘要压缩。

**学到的原理**：

- 大模型API的计费依据是处理的Token总数。每轮对话，我都需要把 `system消息 + user1 + assistant1 + user2 + assistant2 + ... + 当前user消息` 全部发给模型，所以随着对话轮次增加，**每次请求的输入Token数会线性增长**。
- 举例：如果每轮对话平均消耗200 Token，到第50轮时，单次请求的输入Token可能已经上万，费用是初始的几十倍。
- 这是大模型应用开发面临的**第一个成本控制问题**，也是面试中常问的工程实践题。

**解决方案总结**：

- **滑动窗口裁剪**：只保留最近N轮对话，早期对话直接丢弃。
- **Token阈值裁剪**：用 `tiktoken` 库实时计算 `messages` 的Token总数，超过阈值时自动删除最早的几轮。
- **摘要压缩**：对早期对话内容做自动摘要，保留关键信息的同时大幅减少Token占用。

### 问题8：401 Unauthorized vs 403 Forbidden——两个“拒绝访问”的区别

**发现过程**：在学习HTTP状态码时，我最初只知道403是“禁止访问”，401则不太清楚。后来在API调用中也确实遇到了认证失败的情况。

**学到的原理**：

| 状态码  | 名称         | 含义                           | 类比                                         | 大模型API场景                                 |
| ------- | ------------ | ------------------------------ | -------------------------------------------- | --------------------------------------------- |
| **401** | Unauthorized | “你是谁？请出示证件”           | 你想进大楼，但根本没出示门禁卡               | API Key缺失、错误、或格式不对                 |
| **403** | Forbidden    | “你的身份我清楚，但你没有权限” | 你出示了门禁卡，但你的卡没有进入这层楼的权限 | API Key有效，但没有调用该模型的权限或余额不足 |

**排查方法**：
- 遇到401：检查Key是否正确、是否过期、Header格式是否为 `Bearer sk-xxx`、Base URL是否指向正确的服务商。
- 遇到403：检查账户权限、模型访问权限、套餐是否支持该接口。

## 三、最终产出：可运行的终端多轮对话程序

**核心逻辑**：
1. 加载 `.env` 中的 API Key
2. 创建 `client`（openai库）或直接写 `sendMessage` 函数（requests库）
3. 初始化 `messages` 列表，可选加一条 `system` 角色消息
4. `while True` 循环接收用户输入
5. 每次输入后，将用户消息加入 `messages`，调用API
6. 从响应中提取助手内容，再将其加入 `messages`
7. 输入 `exit` 退出循环

**两种发送函数**：
- `sendMessage(messages)`：基于 `requests.post`，手动处理URL、Header、Body，返回字典
- `sendChat(messages)`：基于全局 `client` 对象的 `chat.completions.create`，返回对象
