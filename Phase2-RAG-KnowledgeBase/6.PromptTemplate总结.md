## 二、全局视角：LangChain 中的提示词模板有哪些种类

在深入细节之前，我先建立全局认知。LangChain 里的提示词模板从功能形态上分为四类：

| 模板类型                              | 核心作用                                    | 典型场景                                       |
| :-------------------------------- | :-------------------------------------- | :----------------------------------------- |
| **字符串模板** `PromptTemplate`        | 单条纯文本提示，变量替换后生成一个字符串                    | 直接传给 LLM（补全模型），如早期 text-davinci            |
| **聊天模板** `ChatPromptTemplate`     | 组合多条消息（system / human / ai），生成消息列表      | 对接 Chat Model，如 GPT-4、DeepSeek-Chat，这是目前主流 |
| **少样本模板** `FewShotPromptTemplate` | 动态拼接示例到 prompt 中，实现 in-context learning | 希望模型照着范例格式输出                               |
| **管道模板** `PipelinePromptTemplate` | 把多个子模板拼装成一个最终 prompt                    | 分角色、分片段复杂组装                                |

它们的继承关系如下：

```text
BasePromptTemplate（所有模板的根）
├── StringPromptTemplate
│   ├── PromptTemplate          ← 我第一阶段学习的重点
│   └── FewShotPromptTemplate  ← 继承自字符串模板，额外注入示例
└── ChatPromptTemplate         ← 返回的不是字符串，而是消息列表
```

**我的认知定位**：当前阶段，我的核心学习对象是 `PromptTemplate`（打基础）和 `ChatPromptTemplate`（对接主流 Chat Model）。本文档聚焦 `PromptTemplate` 的完整掌握。

## 三、`PromptTemplate` 核心机制

### 3.1 两种创建方式

`PromptTemplate` 有两种实例化方式。

**方式一：`PromptTemplate.from_template()`（推荐）**

```python
from langchain_core.prompts import PromptTemplate

template = "请将以下文本翻译成{target_language}：{text}"
prompt = PromptTemplate.from_template(template)
```

`from_template` 是类方法，内部会**自动提取**花括号内的变量名（这里是 `target_language` 和 `text`），无需我手动声明 `input_variables`。

**方式二：构造函数** **`PromptTemplate()`**

```python
prompt = PromptTemplate(
    template="请将以下文本翻译成{target_language}：{text}",
    input_variables=["target_language", "text"]
)
```

这种方式我必须手动传入 `input_variables`，更精确但也更繁琐。

**类比**：方式一就像 Java 里用 Lombok 的 `@AllArgsConstructor`，自动推导；方式二则是手动写全参构造器。

***

### 3.2 参数详解

`PromptTemplate` 的完整参数列表如下：

| 参数                  | 类型                 | 必填                      | 作用                                      |
| :------------------ | :----------------- | :---------------------- | :-------------------------------------- |
| `template`          | `str`              | ✅ 是                     | 提示词模板字符串，用 `{变量名}` 占位                   |
| `input_variables`   | `List[str]`        | 否（`from_template` 自动提取） | 声明模板中出现的变量名                             |
| `partial_variables` | `Dict[str, Any]`   | 否                       | 预设部分变量的默认值，`format`/`invoke` 时可省略，传入则覆盖 |
| `input_types`       | `Dict[str, str]`   | 否                       | 变量类型映射，影响某些模型的特殊行为                      |
| `output_parser`     | `BaseOutputParser` | 否                       | 输出解析器，在链式调用时传递给下游                       |

**`partial_variables`** **的实际用法**：

```python
template_str = "你好，{name}！今天天气{weather}。"
prompt = PromptTemplate.from_template(
        template_str,
        partial_variables={
            "weather": get_current_weather,   # 可以传函数，调用时动态求值
        }
    )
    print(prompt.format(name="李华"))  # weather 会自动调用函数
    # 覆盖 partial_variables 的值
    print("\n6. 二次填充可覆盖 partial_variables 的值")
    print(prompt3.format(name="李华", weather="暴雨"))  # 这里 weather 被覆盖为 "暴雨"
```

这个特性非常适合封装公共配置，比如目标语言、输出格式、角色设定等。

### 3.3 `format` 与 `invoke` 两种核心操作

这是我在学习过程中花费最多时间理清的概念，也是面试高频考点。

**`format()`** **—— 返回普通字符串**

```python
result = prompt.format(target_language="英语", text="你好，世界！")
print(type(result))  # <class 'str'>
print(result)        # 请将以下文本翻译成英语：你好，世界！
```

**`invoke()`** **—— 返回** **`PromptValue`** **对象**

```python
prompt_value = prompt.invoke({"target_language": "英语", "text": "你好，世界！"})
print(type(prompt_value))
# <class 'langchain_core.prompt_values.StringPromptValue'>
```

`StringPromptValue` 是 LangChain 生态的一等公民，它有两个关键转换能力：

```python
prompt_value.to_string()    # 转字符串 → "请将以下文本翻译成英语：你好，世界！"
prompt_value.to_messages()  # 转消息列表 → [HumanMessage(content="请将以下文本翻译成英语：你好，世界！")]
```

| 对比维度          | `format`              | `invoke`               |
| :------------ | :-------------------- | :--------------------- |
| 返回类型          | `str`                 | `StringPromptValue`    |
| 对接 LLM（补全模型）  | ✅ 直接可用                | 需 `to_string()`        |
| 对接 Chat Model | ❌ 需手动包 `HumanMessage` | ✅ `to_messages()` 自动转换 |
| 接入 LCEL 链     | ❌ 不支持                 | ✅ 可直接流入下一个组件           |
| 适用场景          | 调试、打印、旧代码兼容           | **所有新项目**              |

**类比**：`format` 就像直接用 `StringBuilder` 拼 SQL 字符串，能跑但不安全难维护；`invoke` 则是用 JPA 的 `Specification`，让框架管理类型和转换。

## 四、Chat Model 对接：格式转换的本质

### 4.1 Chat Model 需要什么？

`ChatDeepSeek`、`ChatOpenAI` 这些 Chat Model 对象，本质上是 HTTP 客户端的封装：

```python
chat_model = ChatDeepSeek(
    model="deepseek-chat",
    api_key=os.getenv("DEEPSEEK_API_KEY")
)
```

它们帮我们做了三件事：

| 配置项        | 作用                  | Java 类比                    |
| :--------- | :------------------ | :------------------------- |
| `api_key`  | 认证头 `Authorization` | `RestTemplate` 的请求拦截器      |
| `base_url` | API 服务地址            | `RestTemplate` 的 `baseUrl` |
| `model`    | 请求体中的模型名            | 请求 JSON 的 `"model"` 字段     |

Chat Model 的 `invoke()` 方法要求传入的是一个**消息列表**：`[HumanMessage(...)]` 或 `[SystemMessage(...), HumanMessage(...)]`。

**`PromptTemplate.format()`** **返回的** **`str`，不能直接传给 Chat Model**，因为裸字符串不是消息列表。

***

### 4.2 `format` 的手动转换路径

```python
from langchain_core.messages import HumanMessage

template_str = "你好，{name}！今天天气{weather}。"

prompt = PromptTemplate.from_template(template_str)
prompt_value = prompt.format(name="李华", weather="暴雨")
print("format 返回的类型：", type(prompt_value))

message = HumanMessage(content=prompt_value)
print("转换后message 类型：", type(message))

# 调用模型
result = chat_model.invoke(message)
print(result.content)
```

**多轮对话时，我手动管理消息列表**：

```python
from langchain_core.messages import HumanMessage, AIMessage, SystemMessage

history = [
    SystemMessage(content="你是一个翻译助手。"),
    HumanMessage(content="翻译：你好"),
    AIMessage(content="Hello"),
]

new_str = prompt.format(target_language="英语", text="谢谢")
new_msg = HumanMessage(content=new_str)

all_messages = history + [new_msg]                 # 手动拼接历史

result = chat_model.invoke(all_messages)
```

***

### 4.3 `invoke` 的自动转换路径

```python
prompt_value = prompt.invoke({"target_language": "英语", "text": "你好"})
result = chat_model.invoke(prompt_value)            # 直接传入，自动转换
```

`chat_model.invoke()` 内部会调用 `prompt_value.to_messages()`，自动转成 `[HumanMessage(...)]`。

**更优雅的方式——LCEL 链式调用**：

```python
chain = prompt | chat_model
result = chain.invoke({"target_language": "英语", "text": "你好"})
```

`PromptValue` 在管道中自动流动，完全不需要手动处理格式。

***

## 五、单轮与多轮对话的本质

### 5.1 模型是无状态的

DeepSeek、GPT 等大模型的 API **不保存任何对话历史**。每次调用都是独立的，模型只根据当前请求中的 `messages` 列表来生成回复。

**单轮对话**：请求体只有 1 条 user 消息。\
**多轮对话**：请求体中包含多条消息（user → assistant → user → ...），模型通过完整的历史上下文来理解“那儿的”、“它”等指代关系。

**两者的 API 调用方式完全一样**，区别仅在于 `messages` 数组的长度。

***

### 5.2 `PromptTemplate` 的职责边界

`PromptTemplate` 是**单条消息内容的生成器**，它不关心对话有几轮。无论是单轮还是多轮，它只负责生成当前这一轮的用户消息文本。

- 单轮场景：`format` 或 `invoke` 生成的文本，包装成单条消息直接发送。
- 多轮场景：生成的文本，需要**我自己**把它和历史消息拼接成一个列表再发送。

**`PromptTemplate`** **不能管理多轮对话的消息结构**。这个能力边界，直接引出了对 `ChatPromptTemplate` 的需求。

***

## 六、最终产出

本次学习，我产出的核心认知与可交付技能：

1. **全局认知**：LangChain 提示词模板有四类，当前阶段核心是 `PromptTemplate` 和 `ChatPromptTemplate`。
2. **两种创建方式**：`from_template()`（自动提取变量，推荐）和直接构造函数（手动声明变量）。
3. **参数清单**：`template`（必填）、`input_variables`、`partial_variables`、`input_types`、`output_parser`。**没有** **`output_variables`**。
4. **`format`** **vs** **`invoke`**：前者返回 `str`（调试用），后者返回 `StringPromptValue`（生产用，支持 LCEL 链）。
5. **Chat Model 对接**：`format` 的结果必须手动包 `HumanMessage`；`invoke` 的结果可自动转换，直接传入。
6. **多轮对话本质**：模型无状态，多轮能力来自客户端维护的消息列表。`PromptTemplate` 只生成单条消息内容，不管理列表结构。

## 七、面试高频考点

| 考点                               | 对应知识点           | 我的回答思路                                                                                                                |
| :------------------------------- | :-------------- | :-------------------------------------------------------------------------------------------------------------------- |
| `PromptTemplate` 的参数有哪些？         | 模板、变量声明、部分变量    | 必填 `template`；`input_variables` 可被 `from_template` 自动提取；`partial_variables` 预设可覆盖；特别注意 **没有** **`output_variables`**。 |
| `format()` 和 `invoke()` 的区别？     | 返回值类型、LCEL 链式调用 | `format` 返回 `str`，旧方式，不适用于链式调用。`invoke` 返回 `PromptValue`，LangChain 标准件，支持 `\|` 管道符，是未来。                               |
| 为什么 `invoke` 的结果能直接给 Chat Model？ | 多态、类型转换         | `StringPromptValue` 实现了 `PromptValue` 接口的 `to_messages()` 方法。Chat Model 内部会调用该方法完成自动转换。                               |
| 如何实现多轮对话？                        | 模型无状态、消息历史      | 模型不记忆，客户端需维护 `messages` 列表，每次请求发送完整历史。`PromptTemplate` 只负责生成当前轮消息内容。                                                  |

## 八、个人成长小结

这次学习让我从“会用”到“会道”，从单纯跑通代码到理解组件间的**类型对接和职责边界**。

- **我掌握了**：`PromptTemplate` 从创建到格式化的完整生命周期，以及它如何通过 `invoke` 接入 LCEL 生产链路。
- **认知跃迁**：LLM 应用开发的核心不是发请求，而是**数据流的编排与转换**。`PromptTemplate` 输出 `str` 还是 `PromptValue`，决定了它与下游组件的兼容性。这种对类型和接口的敏感度，是我从后端开发迁移过来的宝贵经验。
- **避坑指南**：
  1. **忘记** **`output_variables`**：它不存在于 `PromptTemplate` 中。
  2. **新项目拥抱** **`invoke`**：只要用 LangChain 和 LCEL，就以 `invoke` 为入口。
  3. **`PromptTemplate`** **不背多轮的锅**：管理聊天历史结构是 `ChatPromptTemplate` 的职责，不要试图用 `PromptTemplate` 硬塞。

我现在的知识版图已经清晰地指向下一步：**学习** **`ChatPromptTemplate`** **来处理多角色、多轮对话的模板构建**，这是构建实用聊天机器人的必经之路。
