# 一、初始知识点

> 我对 Chain 的认知不是零起步，而是在看完了 P06 的 RAG 架构后，带着"这个东西为什么要有这么多层封装"的疑问进来的。学之前，我已经能基于 `requests` 手写 OpenAI API 调用，也理解了 Prompt、LLM、Parser 三者可以拆开单独调试。

在开始本次学习之前，我手里已经攒了这些砖头：

1. **管道的直觉**\
   我知道 RAG 的流程是"检索文档 → 塞进 Prompt → 给大模型 → 解析答案"，如果每一步都自己写函数传参，代码会特别乱。所以我本能地需要一种能**把多个步骤串起来**的工具。
2. **LLM 与 Prompt 的基本配合**\
   我已经能写出 `ChatPromptTemplate` 模板，也知道 LLM 的 `invoke` 接收一个格式化后的字符串，返回一个 `AIMessage` 对象。这让我后续理解 `LLMChain` 时，只是觉得"它不过是帮我把这两步绑一起"，没什么神秘感。
3. **对"传统"和"新式"的模糊感知**\
   看教程时注意到同一个功能居然有新旧两套写法——比如 `StuffDocumentsChain` 和 `create_stuff_documents_chain`，但当时我只觉得后者是前者的"快捷方式"，并没意识到底层整个设计哲学都变了。
4. **一次失败的 Sequencial Chain 调试经历**\
   我自己尝试用 `SimpleSequentialChain` 跑通一个"翻译→总结"的流程时，发现第二个 Chain 只能拿到一个笼统的 `input` 变量，我想传递"源语言"这个额外信息却死活传不过去。这让我隐约感觉到，**Chain 之间的变量传递是有严格规则的**。

## 二、学习过程中遇到的问题 & 如何解决的

> 这一轮学下来，真正的成长不在于记住了多少 Chain 的名字，而在于我把"串行传递数据"这件小事从直觉变成了可控的工程操作。以下四个问题，每条都对应一次认知的断裂和重建。

### 问题 1：`LLMMathChain` 是让大模型直接算数学吗？

- **发现过程**\
  看教程目录时，`LLMMathChain` 被归类在"传统 Chain"下，描述是"专门处理数学问题"。我下意识认为，它是给 LLM 换了一套更适合数学的 Prompt，让它直接算。直到我被提醒"大模型算小数乘法都会翻车"，才发现自己想得太简单。
- **我的原始思考**\
  我觉得模型既然能写代码，那做个数学题应该也一样吧，无非是数学题目的 token 组合更确定一点。
- **学到的原理 / 迭代后的认识**\
  大模型本质是**下一个 token 的概率预测器**，而不是计算器。`237 × 891` 这类精确数值计算，靠 token 概率推演很容易出错。`LLMMathChain` 的解法非常工程化：它让 LLM 把自然语言题目转成 Python 代码字符串，然后**实际调用 Python 解释器执行代码**，拿到硬算出来的数值结果，再格式化返回。这本质上是一个"自然语言 → 代码 → 数值"的编译过程。
- **解决方案总结**\
  以后再遇到需要精确计算的任务，我会优先考虑"让 LLM 生成执行代码，再实际去跑"，而不是死磕 Prompt 去提高它直接算对的概率。这同时也让我对即将学到的 Agent 中的 `Tool` 有了提前的感知——`LLMMathChain` 内部就是靠一个 Python REPL Tool 来完成计算的。

### 问题 2：`SequentialChain` 里，我怎么把字典里某一个字段单独传给下一个 Chain？

- **发现过程**\
  我自己写"翻译→总结"管线时，第一个 Chain 输出了一个字典 `{"translated_text": "...", "source_lang": "zh"}`，但第二个 Chain 的 Prompt 只需要 `translated_text`，不需要 `source_lang`。我不知道该怎么在传递时过滤字段。
- **我的原始思考**\
  我打算在中间写一个解析函数，手动把字典拆开再拼回去。但这就把 Chain 的自动传递优势毁了。
- **学到的原理 / 迭代后的认识**\
  `SequentialChain` 的 `output_key` 是**显式白名单**。你在定义第一个 `LLMChain` 时，必须把想传递的键名写进 `output_key` 列表。没写进去的键，即使 LLM 输出了，也会在内存里被直接丢弃，后续 Chain 不可见。\
  而第二个 Chain 通过 `input_key` 指明接收哪个变量，这个变量名必须和上游 `output_key` 完全对齐，**不是靠 prompt 里的占位符名字自动匹配的**。
- **解决方案总结**\
  在我那个场景下，只需把第一个 Chain 的 `output_key` 设为 `["translated_text"]`，第二个 Chain 的 `input_key` 也设为 `"translated_text"`，就能干干净净地只传一个字段，`source_lang` 被自动忽略。

### 问题 3：`create_stuff_documents_chain` 的 prompt 里必须有一个叫 `context` 的变量吗？

- **发现过程**\
  我看教程的示例代码，prompt 模板里总有个 `{context}`，我以为这只是一个约定俗成的名字，随便改一个也行。但当我在自己写的 prompt 里把 `{context}` 改成 `{docs_list}`，然后用 `invoke({"docs_list": docs, "question": "..."})` 去调用，直接报了 KeyError。
- **我的原始思考**\
  我以为是 `ChatPromptTemplate` 的变量名在起作用，只要我传参时字典的键和占位符名字一样就能跑通。
- **学到的原理 / 迭代后的认识**\
  翻源码才发现，`create_stuff_documents_chain` 内部有一段逻辑是 `docs = inputs.pop("context")`，它**硬编码了从 inputs 字典里取出文档的那个键名就叫** **`"context"`**。这跟你 prompt 占位符叫啥没关系，是调用时传参的字典必须带 `"context"` 这个键，而 prompt 模板也必须用 `{context}` 来接收填入的文档文本。这是当前版本 (langchain 0.3.x) 的实现细节。
- **解决方案总结**\
  以后再写 `create_stuff_documents_chain`，我会严格遵守"调用时传 `{"context": docs}`，prompt 里写 `{context}`"。同时我也知道了，如果未来版本把这个键名改成可配置的，那么内部 `pop` 的键和 prompt 占位符名字必须同时改，且保持一致。

### 问题 4：新旧 Chain 到底有什么区别？为什么教程总说"新的更好"？

- **发现过程**\
  我在浏览 API 文档时，发现 `StuffDocumentsChain` 和 `create_stuff_documents_chain` 都在，新教程却全部用后者。我就很奇怪：既然功能一样，搞两套干嘛？
- **我的原始思考**\
  我以为新版本的 `create_...` 函数只是一个语法糖，底层还是调旧 `Chain` 类，只是换个写法让新手更好上手。
- **学到的原理 / 迭代后的认识**\
  这不是语法糖，是**底层协议的彻底升级**。
  - 传统 Chain 基于 `Chain` 基类，通过 `__call__` 和 `run()` 串联，传递靠字符串拼接，不支持流式输出和自动并行。
  - 新式 Chain 基于 **LCEL（LangChain Expression Language）**，所有组件都是 `Runnable` 对象，用 `|` 管道符连接。\
    只要是 `Runnable`，就可以直接用 `invoke()`（一次性调用）、`stream()`（流式输出）、`ainvoke()`（异步调用），而且 LangSmith 能自动追踪每一步的延迟。`create_stuff_documents_chain` 返回的就是这样一个 `Runnable`，所以它**天然就是可组合的管道组件**，而不是一个封闭的"链"。
- **解决方案总结**\
  从今天起，我脑中的模型要从"Chain 是一串固定步骤"切换成"Chain 是由 Runnable 节点构成的 DAG"。这样我到后面学 Agent 时，把 retriever、prompt、tool 用 `|` 自由拼装，就不会有心理障碍。

## 三、最终产出

> 以下是我今天学完后，能直接拿出来写代码、讲给同事听的硬通货。

1. **能独立对比 5 种传统 Chain 的适用场景与数据流**
   - `LLMChain`：最基础的 Prompt + LLM 绑定
   - `SimpleSequentialChain`：单输出接力，固定 `input`
   - `SequentialChain`：多输出接力，`output_key` 和 `input_key` 显式对齐
   - `LLMRouterChain`：根据输入动态选择下游 Chain
   - `StuffDocumentsChain`：多文档拼接成单 prompt，适合总结
2. **理解并会解释** **`LLMMathChain`** **的底层机制**\
   能说出它是"自然语言 → Python 代码 → 数值结果"的编译执行链路，知道为什么这样做能避免大模型的数值错误。
3. **掌握了** **`SequentialChain`** **的变量传递白名单规则**\
   知道 `output_key` 是上游选择性暴露字段的唯一手段，没写的字段直接被丢弃；下游 `input_key` 必须完全匹配。能在代码层面正确实现多字段选择传递。
4. **搞懂了** **`create_stuff_documents_chain`** **的强制** **`context`** **约定**\
   知道内部代码 `pop("context")` 是硬编码的，调用时传参字典和 prompt 模板都必须用 `"context"` 这个名字，与占位符自定义无关。
5. **建立 LCEL 心智模型**\
   明确新旧 Chain 的核心差异在于 `Runnable` 协议，`create_...` 系列返回的都是可组合的 `Runnable` 组件，统一支持 `invoke/stream/ainvoke`，后续 RAG 和 Agent 的拼装全部基于这个协议。

## 五、个人成长小结

> 今天我不再觉得 Chain 只是"省事儿的函数封装"了。我意识到，它是一种**把大模型应用从脚本思维提升到工程架构思维的分界线**。

我最大的认知跃迁在于：**控制数据流的显式性和可预期性，远比"代码短"更重要**。

- `output_key` 是白名单，不是装饰——它决定了下游能看到什么，多一个少一个都可能断链。
- `create_stuff_documents_chain` 的 `context` 硬编码，不是"麻烦"，而是明确的设计约束，让我知道系统边界在哪。
- LCEL 的 `Runnable` 协议，不是"新语法糖"，而是让我的项目可以直接上生产（流式、异步、可追踪）的底层保障。

这些认知让我在接下来做"校园助手全栈交付"时，不会再眉毛胡子一把抓地写脚本，而是会先用纸笔画出 Runnable 之间的管道图，再动手写代码。

**今日避坑指南**

1. **别把** **`SimpleSequentialChain`** **当万能胶**：只要下游需要多个输入变量，就立刻换 `SequentialChain`，否则多余信息全部丢失。
2. **`output_key`** **写不全，debug 到崩溃**：上游输出的字典有 5 个字段，你只写了 3 个进 `output_key`，后面别想找到另外 2 个。
3. **`create_stuff_documents_chain`** **不要改** **`context`** **名字**：调用字典键和 prompt 占位符都必须老老实实叫 `context`，除非你确认自己能够修改源码适配且清楚潜在影响。
4. **别拿** **`LLMMathChain`** **当计算器用**：它适合解数学应用题，简单的四则运算直接用 Python 算更快更准，别非过一遍 LLM。
