# ChatPromptTemplate 核心机制学习笔记

## 一、初始知识点

在正式学习 `ChatPromptTemplate` 之前，我已完全掌握了 `PromptTemplate` 的以下核心概念：

- **两种创建方式**：`PromptTemplate.from_template()` 和 `PromptTemplate()` 构造函数。
- **两种核心操作**：`format()` 返回 `str`；`invoke()` 返回 `StringPromptValue`，能通过 `to_messages()` 自动转为 `[HumanMessage(...)]`。
- **职责边界**：`PromptTemplate` 是单条消息内容的生成器，不关心角色，不管理消息列表结构。

---

## 二、学习过程中遇到的问题 & 如何解决的

### 问题 1：`ChatPromptTemplate` 和 `PromptTemplate` 的根本区别

- **发现过程**：在尝试构建需要 system 和 human 两条消息的提示词时，我发现 `PromptTemplate` 无法胜任，强行使用会导致角色信息丢失。
- **我的原始思考**：我最初认为 `ChatPromptTemplate` 只是比 `PromptTemplate` 多了一个消息列表，两者本质相似。
- **学到的原理 / 迭代后的认识**：它们的本质区别在于**职责维度不同**。
    - `PromptTemplate` 的 `invoke()` 返回 `StringPromptValue`，只对应**一条** `HumanMessage`。
    - `ChatPromptTemplate` 的 `invoke()` 返回 `ChatPromptValue`，内部是一个**多角色消息列表**（可以包含 system、human、ai 等）。
    - 两者都实现了 `PromptValue` 接口，都能在 LCEL 链中与 Chat Model 无缝对接。
- **解决方案总结**：当模板只包含一条用户消息时，两者可互替；但只要涉及 system 提示或多角色，就必须用 `ChatPromptTemplate`。

---

### 问题 2：`from_messages` 中的元素类型，以及字符串列表的误区

- **发现过程**：在回答面试题时，我错误地判断“全是字符串的列表”也能正确构建带 system 角色的模板。
- **我的原始思考**：我误以为 `ChatPromptTemplate.from_messages(["系统提示", "用户问题"])` 能自动识别第一句是系统指令。
- **学到的原理 / 迭代后的认识**：通过打印 `format_messages()` 的结果，我亲眼看到两条字符串都被转成了 `HumanMessage`，system 角色完全丢失。
    - **`from_messages` 的适配规则**：
        1.  **字符串** → 自动转为 `HumanMessagePromptTemplate`。
        2.  **双元素元组** `("role", "content")` → 根据第一个元素映射到对应角色（system/human/ai）。
        3.  **消息对象**（如 `HumanMessage`） → 原样保留。
    - 因此，要构建多角色模板，**必须使用元组或消息对象来明确声明角色**。
- **解决方案总结**：标准写法是 `[("system", "..."), ("human", "...")]`。面试中若把纯字符串列表当正确写法，会直接暴露对角色机制的误解。

---

### 问题 3：四种填充方法何时用

- **发现过程**：`ChatPromptTemplate` 提供了 `format`、`format_prompt`、`format_messages`、`invoke` 四种方法，一开始让我觉得眼花缭乱。
- **我的原始思考**：我认为它们功能重叠，可能只是不同版本的遗留接口。
- **学到的原理 / 迭代后的认识**：它们各自有精确的使用场景。

| 方法 | 返回类型 | 使用场景 |
| :--- | :--- | :--- |
| `format(**kwargs)` | `str` | 调试、打印，快速查看整个对话模板的文本全貌。 |
| `format_prompt(**kwargs)` | `ChatPromptValue` | 功能同 `invoke`，但不兼容 LCEL，属于早期接口。 |
| `invoke(dict)` | `ChatPromptValue` | **LCEL 链式调用的标准入口**，生产环境首选。 |
| `format_messages(**kwargs)` | `List[BaseMessage]` | 需要拿到原始消息列表，进行追加、修改或打印。 |

- **解决方案总结**：新项目统一用 `invoke` 对接 LCEL；需要手动处理消息列表时用 `format_messages`；调试看文本用 `format`。`format_prompt` 基本可忽略。

---

### 问题 4：如何优雅地动态注入聊天历史——`MessagesPlaceholder`

- **发现过程**：我想构建一个能记住上下文的聊天机器人。如何把一段动态长度的历史消息，注入到一个固定的模板里？
- **我的原始思考**：我设想在 `invoke` 之前手动做列表拼接：`history + new_messages`。这能工作，但让模板变得不完整，逻辑散落。
- **学到的原理 / 迭代后的认识**：`MessagesPlaceholder` 提供了一个**声明式占位符**，完美解决此问题。
    - **声明占位**：在 `from_messages` 中声明 `MessagesPlaceholder(variable_name="history")`。
    - **运行时注入**：在 `invoke({"history": [...], "input": "..."})` 时，直接把历史消息列表作为变量传入。它在模板中的位置会被自动展开。
    - **`optional=True` 参数**：如果一个消息块（如可选的系统提醒）是可选的，设置 `optional=True`，调用时不传该变量也不会报错。
- **解决方案总结**：这是构建任何有记忆的聊天应用的核心模式，让模板定义和状态管理解耦。

## 三、最终产出

本次学习，我产出的核心认知与可交付技能如下：

1.  **明确职责区分**：
    - `PromptTemplate` → 单条 `HumanMessage`。
    - `ChatPromptTemplate` → 多角色 `[SystemMessage, HumanMessage, ...]` 列表。

2.  **掌握唯一推荐构造方式**：
    - `ChatPromptTemplate.from_messages([...])`：通过元组 `("role", "content")` 明确角色，是标准做法。**禁止用裸字符串列表构建多角色模板**。

3.  **精通四种操作方法**：
    - `invoke(dict)`：LCEL 链式调用标准入口。
    - `format_messages(**kwargs)`：获取 `List[BaseMessage]`。
    - `format(**kwargs)`：调试用，返回纯文本。
    - `format_prompt(**kwargs)`：旧式接口，可忽略。

4.  **掌握动态历史注入**：
    - 使用 `MessagesPlaceholder(variable_name="history")` 在模板中声明历史消息位。
    - 在 `invoke` 时将完整的 `List[BaseMessage]` 历史通过该变量传入。

5.  **形成类型流理解**：
    - `ChatPromptTemplate` 的输出 `ChatPromptValue` 可通过 `.to_messages()` 转列表，`.to_string()` 转文本，与 `StringPromptValue` 共享相同的 `PromptValue` 接口。

## 四、面试高频考点

| 考点 | 对应知识点 | 我的回答思路 |
| :--- | :--- | :--- |
| `PromptTemplate` vs `ChatPromptTemplate` | 职责维度、返回类型 | 一个管单条消息，一个管多角色消息列表。`invoke` 返回的 `PromptValue` 子类不同，但都能对接 Chat Model。 |
| `ChatPromptTemplate` 的构造方式 | `from_messages`，元素类型 | 使用 `from_messages` 并用元组 `("role", "content")` 明确角色。字符串列表会丢失角色信息，变成全是 `HumanMessage`。 |
| 如何动态插入聊天历史？ | `MessagesPlaceholder` | 在模板中声明占位符 `MessagesPlaceholder(variable_name="history")`，运行时通过 `invoke` 传入历史列表。可设置 `optional=True` 处理可选情况。 |
| `invoke`, `format`, `format_messages` 的区别？ | 返回类型、适用场景 | `invoke` 返回 `ChatPromptValue`，用于 LCEL；`format_messages` 返回 `List[BaseMessage]`，用于手动处理；`format` 返回 `str`，用于调试。 |

## 五、个人成长小结

通过这次学习，我补齐了从“字符串模板”到“结构化对话模板”的关键拼图。

- **技能层面**：我能够熟练地用 `ChatPromptTemplate.from_messages` 构建多角色指令，并利用 `MessagesPlaceholder` 为后续实现带记忆的对话机器人打下了坚实基础。
- **认知跃迁**：我理解了 **“适配器模式”在 LangChain 中的体现**——`from_messages` 将字符串、元组、消息对象等多种格式统一转化为内部标准模板，这比 `PromptTemplate` 的纯字符串视角更复杂、也更强大。我明白了结构化的 `messages` 列表才是现代聊天模型的“母语”。
- **避坑指南**：
    1.  **别用裸字符串列表**：`["sys", "human"]` 这种写法会丢失系统角色，必须用元组。
    2.  **拥抱 `invoke`**：忘记 `format_prompt`，一切以 `invoke` 为入口。
    3.  **历史管理用 `MessagesPlaceholder`**：它是管理动态消息列表的唯一标准方式，别再想着手动拼接字符串到模板里。

我现在已经掌握了 RAG 流程中“输入端”的两大核心模板工具。下一步，就是将这些结构化的提示词，与**文档加载、文本切分、向量数据库**串联起来，真正跑通一个 RAG 问答流程。
