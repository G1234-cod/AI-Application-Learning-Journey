## 📝 今日核心认知

> 学习时间：2026.05 
>
> **RAG 和 Agent 不是“二选一”的互斥方案，而是解决不同维度问题的分层架构。RAG 负责给模型“喂”私有知识，Agent 负责给模型“装”手脚去规划执行。最核心的认知跃迁是：不要拿着“向量化”的锤子，看什么都是钉子——不是所有外部信息获取都需要走向量检索这条路。**

---

## 📚 知识点清单

### 1. RAG 与 Agent 的正确关系（与 Java 后端知识对接）

-   **发现过程**：我学完 RAG 架构总览后，本能地把 AI 应用开发分为“RAG 型”和“Agent 型”两大类，认为它们是平行的两种方案。以为做知识问答就用 RAG，做自动化任务就用 Agent，两者泾渭分明。

-   **迭代后的认识**：
    
    -   **关系**：RAG 和 Agent **不是互斥的，而是分层、可组合的**。RAG 是一种“知识注入手段”，Agent 是一种“自主规划范式”。Agent 可以把 RAG 当作工具箱里的一件工具来调用。
    -   **Java 类比**：RAG 像 MyBatis 的 Mapper 层，专门负责从外部数据源查数据；Agent 像 Spring 的 `@Service` 编排层，决定调用哪个 Mapper、哪个第三方 API，然后组装结果。
    -   **核心区别表**：
        | 维度     | RAG                                | Agent                                          |
        | -------- | ---------------------------------- | ---------------------------------------------- |
        | 核心职能 | “百科顾问”，回答基于特定文档的问题 | “全能助理”，自主规划步骤、调用工具             |
        | 工作流   | 线性：检索→增强→生成               | 循环：思考→行动→观察→再思考                    |
        | 解决问题 | 知识滞后、私有数据问答、幻觉控制   | 需要多步推理、实时信息获取、执行动作的复杂任务 |
    -   **“合体”场景**：一个校园助手 Agent，工具箱里可以同时放一个 RAG 模块（查校内规章制度）和一个联网搜索 API（查外部新闻），Agent 自己判断什么时候用谁。

---

### 2. LLM 应用范式的四层递进（纠正了我对“上网搜索”和 MCP 的误用）

-   **发现过程**：我基于自己的理解，归纳了“纯 Prompt、Prompt+RAG、Prompt+Agent、Prompt+Agent+RAG”四种情况，但在描述中把“上网搜索”直接等同于 RAG，还误用了 MCP 这个术语。

-   **迭代后的认识**：
    -   **RAG ≠ 上网搜索**：RAG 检索的是**我提前喂进去的、存在本地向量库里的私有文档**（如 PDF、TXT）。上网搜百度、Google，那是 Agent 调用“联网搜索工具”的能力，和 RAG 是两套完全不同的机制。
    -   **MCP 不是搜索协议**：MCP（Model Context Protocol）是连接模型与工具的通信协议标准，像 USB 接口一样，是用来规范 Agent 怎么调用工具的。
    -   **修正后的能力分层图**：
        
        ```
        Agent（自主决策、规划、工具调用）
           │
           ├── 工具1：RAG（查本地私有知识库）
           ├── 工具2：联网搜索 API（查互联网）
           └── 工具3：计算器等
           │
        基础能力层
           ├── 纯 Prompt（参数化知识，直接回答）
           └── RAG（独立使用，注入外部文档）
        ```
    
-   **解决方案总结**：以后描述时，我会说“RAG 注入私有知识，Agent 调用外部工具”，而“上网搜索”是 Agent 工具箱里的一个选项，不是什么情况都得先向量化。

---

### 3. 向量化不是万能钥匙：RAG 与联网搜索工具的决策边界

-   **发现过程**：对于面试题“问校长最新讲话 vs. 问校长是谁”，让我彻底暴露了错误思维。我回答两个问题都要“先将提示词转为向量”，结果一个都不对。我以为只要涉及外部信息，第一步都得把用户提问向量化，然后去库里搜。我把向量检索当成了获取所有外部信息的唯一入口。

-   **迭代后的认识**：
    -   **问“最新讲话”**：需要的是**联网搜索**。Agent 直接把提问文本（或优化后的查询词）扔给搜索引擎 API，搜索引擎用自己的倒排索引去互联网抓结果。**跟向量一毛钱关系都没有**。
    -   **问“校长是谁”**：大模型的**参数化知识**很可能已经存了这个静态事实。直接让模型回答是最快、成本最低的方案。动用什么 RAG、Agent，都是**过度设计**。
    -   **决策模型**：
        
        ```text
        用户提问
          ├─ 模型自己能答（常识、静态事实） → 直接生成答案
          └─ 模型答不了
               ├─ 需要我私有文档里的权威知识 → 用 RAG（向量检索本地库）
               └─ 需要最新的、联网才能获取的信息 → 用 Agent 的联网搜索工具
        ```

---

### 4. 搜索结果矛盾的处理策略（面试中的开放性问题）

-   **发现过程**：对于面试题“ Agent 联网搜回来的信息相互矛盾怎么办”。这确实是我没考虑过的工程难点。
-   **迭代后的认识**：可以从简到繁，分三层策略：
    1.  **透明化呈现（基准线）**：把矛盾点、各自来源直接列给用户，让用户自行判断。这是最低成本、最务实的做法。
    2.  **基于可信度排序**：提前给信息源打权重分（如官方 > 自媒体），按高分源下结论。
    3.  **多步验证（高级）**：Agent 感知矛盾后，自主发起第二轮搜索，进行交叉验证，确认事实。

## 🌱 个人成长小结

今天我最大的认知跃迁，是从“会背概念”进化到“知道什么时候该用哪个概念”。我不再简单地把 RAG 和 Agent 当成两个目录，而是构建了一个分层、可组合的能力模型。尤其通过“校长讲话 vs. 校长是谁”这道题的反复打磨，我真正理解了**向量的使用边界**，以及**过度设计是工程中的大忌**。这种“先找知识源头，再选技术路径”的决策思维，会让我在秋招面试中显得更有系统设计感，而不只是个调包侠。

---

## ⚠️ 避坑指南

1.  **别把 RAG 当成“上网搜索”**，面试时混淆这俩会直接暴露对核心概念理解不清。
2.  **别上来就向量化**，获取外部信息不只有向量检索一条路，联网搜索是文本直接调用 API。
3.  **别什么需求都上 RAG 或 Agent**，能靠模型参数回答的静态常识，直接回答最快最省钱。
4.  **现阶段别乱用 MCP 术语**，在没深入学 Agent 和工具调用协议前，先不提，避免露怯。
